Project 01

Warehouse Robot Simulation — Utility-Based Agent

Dijkstra · A* Search · Graph Simulation · 1,440-tick 24-hour model

Design and implementation of a utility-based agent controlling a fleet of robots navigating a warehouse modelled as a weighted graph (10 stations, 15 links) to service customer orders. The agent must minimise average order processing time across a 1,440-tick 24-hour simulation while managing robot capacity (out-tray holds 5 items), order arrival probabilities, and dispatch logic.

Implementation

  • ▸Warehouse graph constructed with networkx (10 stations, 15 weighted links)
  • ▸Dijkstra's algorithm for shortest path calculation between stations
  • ▸A* search for robot-to-order assignment — assigns nearest available robot to each incoming order
  • ▸Dispatch logic: robot navigates to pickup → places in out-tray → returns to dispatch (node 1) when tray full or in-tray empty
  • ▸Metrics tracked: average processing time per order and total completed orders over 24 hours

Pseudocode covered (Part A, joint)

  • ▸Customer order generation with probabilistic arrival
  • ▸Nearest-robot assignment via distance comparison
  • ▸Robot navigation and order servicing logic
  • ▸Driver routine: 1,440-tick simulation loop with metrics

★ What this project taught — uniquely

Graph-based agent simulation with real dispatch constraints. This was the only project in the portfolio where the agent's decisions had to account for physical capacity limits (out-tray = 5) and a continuous-time simulation loop — not just a static optimization. Reasoning about when a robot should break away from its current path to pick up a higher-priority order, versus completing its current delivery, introduced trade-offs that don't appear in stateless classification or optimization problems.

Dijkstra vs A* in a practical context. Both algorithms were implemented and compared on the warehouse graph. A* requires a heuristic — in a warehouse graph with no inherent spatial geometry, choosing an admissible heuristic meant reasoning carefully about what "distance to goal" actually means in the weighted graph. The performance difference on a 10-node graph was small, but the conceptual clarity about when A* adds value over Dijkstra became permanent.

Project 02

Genetic Algorithm — Job Shop Scheduling

Makespan Minimization · Interval Crossover · Swap Mutation · NP-Hard Combinatorics

A Genetic Algorithm for minimizing makespan in a Job Shop Scheduling problem — a classic NP-hard combinatorial optimization problem where J jobs with N sequential operations must be scheduled across M machines to minimize total completion time. The chromosome is a permutation of job indices, with interval-based crossover and swap mutation.

GA components

ComponentDesign
ChromosomePermutation of job indices [1..N]
Population300 random permutations
FitnessInverse makespan (lower = higher fitness)
SelectionRoulette wheel (fitness-proportionate)
CrossoverInterval-based: copy interval from parent1, fill from parent2
MutationSwap mutation, rate = 0.03
Generations100 (default)

Experiments: fewer generations (10–100) and higher mutation rate (30%) tested against default — both degraded solution quality, confirming standard hyperparameter choices.

Tarun's contributions (R2, R4, R5)

  • ▸R2: Pseudocode for computing makespan (fitness) from a chromosome permutation
  • ▸R4 + R5: Function for displaying best vs. random chromosome makespans; coded ComputeMakespan() and random chromosome generation
  • ▸Experiments: varied generations (10–100) and mutation rate (30%) to measure effect on solution quality

★ What this project taught — uniquely

Evolutionary optimization on a combinatorial search space. All other ML projects in this portfolio optimize continuous parameters via gradient descent. Genetic algorithms operate on discrete permutations with no gradient — fitness is computed by simulation, not differentiation. Implementing crossover on permutations (where naive splicing produces invalid schedules) required understanding why interval-based crossover exists and how it preserves feasibility. This is a fundamentally different optimization paradigm.

The NP-hard problem class in practice. Job shop scheduling is NP-hard — no polynomial-time exact algorithm is known. Experiencing this directly (running exact search on small instances, watching it become infeasible as N grows, then seeing the GA find near-optimal solutions efficiently) made the complexity theory concrete in a way it hadn't been from coursework alone.

Project 03

Reinforcement Learning — Grid World Value Iteration

Bellman Equation · Stochastic Navigation · Value Iteration · 4×3 and 10×10 Grids

Implementation of Value Iteration for a stochastic grid world navigation agent. The environment uses an 80-10-10-0 transition model (80% intended direction, 10% each perpendicular, 0% opposite) with discount factor γ=0.99. Implemented and validated on both a 4×3 standard benchmark grid and an extended 10×10 grid.

Key finding: Adding a negative living reward (r=−1) substantially accelerated convergence (22 → 18 iterations) because the agent is incentivised to reach a terminal state quickly, creating sharper value gradients that resolve uncertainty faster.

★ What this project taught — uniquely

The Bellman equation as a working algorithm, not a formula. Every other project in this portfolio learns from data — there is no model of the environment, only examples. Value iteration works from a complete environment model (transition probabilities, rewards, discount factor) and computes the optimal policy by iterating the Bellman equation to fixed point. Implementing this from scratch — not calling a library — made the relationship between the Bellman equation, the discount factor, and convergence rate concrete.

Stochastic transitions and policy robustness. The 80-10-10-0 transition model means the agent cannot simply plan a path — it must compute utilities that account for the probability of accidentally stepping sideways. Seeing how the optimal policy changes when the stochasticity changes (A=0.67 vs A=0.79 vs A=0.80) — and watching convergence speed respond — built intuition for why robust policies in partially controllable environments look different from deterministic planning solutions.
