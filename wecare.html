Overview

The problem with cloud-dependent wearables

In cardiac and fall emergencies, every millisecond matters.

Current consumer wearables send sensor data to the cloud for processing before triggering alerts. In cardiac or fall emergencies, that round-trip introduces delays that can be life-threatening. WECARE eliminates this bottleneck by running the entire detection pipeline on-device — with inference times measured in hundredths of a millisecond.

The system fuses two complementary physiological signals: ECG for continuous cardiac arrhythmia detection and IMU (accelerometer + gyroscope) for real-time fall event classification. Both streams feed into independent compact 1D CNNs whose outputs converge in a shared emergency alert engine deployed via TorchScript.

Critical safety metric: Only 8 false negatives out of 228 fall instances — 96.5% recall on the most safety-critical outcome. Missing a fall is far more dangerous than a false alarm, so the threshold (0.65) was tuned specifically to minimize missed falls rather than maximize precision.

Architecture

Dual-stream 1D CNN on the edge

Both ECG and IMU streams share a 3-block 1D CNN design optimized for edge inference — compact enough to run on microcontroller-class hardware with significant latency headroom remaining.

# Shared 1D CNN architecture (both ECG and IMU streams)
Block 1: Conv1D(in → 64, kernel=7) → ReLU → MaxPool(2)
Block 2: Conv1D(64 → 128, kernel=7) → ReLU → MaxPool(2)
Block 3: Conv1D(128 → 256, kernel=5) → ReLU → MaxPool(2)
Flatten → Dense(128) → Dropout → Dense(n_classes)

# ECG input: 256-sample heartbeat segments at 360 Hz
# IMU input: 100×9 sliding windows (9-channel: 3-axis accel + 3-axis gyro + 3-axis mag)

ECG stream

Dataset: MIT-BIH Arrhythmia Database (PhysioNet) — ~10,000 beat segments, 44 subjects, 360 Hz.

Preprocessing: Bandpass filter (0.5–40 Hz) → R-peak segmentation (256-sample windows) → normalization.

Challenge: ~75% class imbalance (normal beats). Addressed with class weights + WeightedRandomSampler to prevent the model from ignoring arrhythmia beats entirely.

IMU stream

Dataset: MobiFall_processed (Kaggle) — ~40,000 windows, 24 participants, 87–100 Hz.

Preprocessing: Butterworth filter → sliding window (100 samples × 9 channels) → StandardScaler per channel.

Threshold tuning: Decision threshold set at 0.65 rather than the default 0.5. Chosen specifically to minimize missed falls (FN=8) while accepting a higher false positive rate.

Extended vision — AIMS proposal

A grant-style AIMS proposal (available in the repository) outlines the long-term vision: integrating ECG + IMU + PPG, automated emergency contact alerts, and multi-actor coordination with bystanders and telemedicine platforms. The core innovation is a wearable orchestration engine addressing the critical pre-responder window where no existing platform currently operates.

Future Work

What comes next

INT8 quantization for microcontroller deployment (nRF52840, ESP32) — the compact CNN is already close to on-chip feasible. Multi-class ECG expansion beyond binary arrhythmia classification. PPG integration for contactless SpO₂ and heart rate fusion. Adaptive per-user thresholding to reduce individual false positive rates. Free-living pilot studies for real-world robustness validation outside controlled lab conditions.

★ What this project taught — uniquely

Edge AI as a hard constraint, not a preference. Every other project in this portfolio can run on any hardware with enough memory. WECARE had a real deployment target: a wearable device where inference must complete in <40ms, no cloud calls, no RNN overhead. Designing for this constraint from the start — compact 3-block 1D CNN, TorchScript export, no recurrent layers — is a fundamentally different design process than optimizing for accuracy on a benchmark. The 0.033ms result isn't just a number; it's headroom for battery optimization.

Threshold tuning as a clinical safety decision. Setting the IMU fall detection threshold at 0.65 (rather than the default 0.5) was a deliberate clinical trade-off: accept more false positives (alarm fatigue risk) to minimize false negatives (missed falls, which are life-threatening for elderly patients). This is the kind of asymmetric cost reasoning that clinical AI requires — different failure modes have different consequences, and the threshold is a policy decision as much as a technical one.

Class imbalance handling for real biosignal data. ECG data is ~75% normal beats by nature — a model that predicts "normal" for every input achieves 75% accuracy while being clinically useless. Addressing this with class weights + WeightedRandomSampler (rather than just oversampling or augmentation) was the specific solution that preserved the full dataset while correcting the learning objective. This technique directly transfers to any medical imaging dataset with rare condition prevalence.

Deployment format matters: TorchScript over TFLite. The original plan was TFLite deployment, which failed due to dependency conflicts between PyTorch-trained models and TensorFlow Lite's export pipeline. Switching to TorchScript — PyTorch's native deployment format — produced a stable, self-contained model file with no runtime framework dependencies. Understanding the deployment ecosystem (not just model training) and making pragmatic format decisions is a real production skill.
